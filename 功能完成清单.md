## 功能一:聊天支持中文(重写收发编码协议) —— 对应文档要求⑤ 【已完成 ✅】

**问题根源**:参考程序的通信协议是面向单字节 ASCII 设计的,从两个层面破坏中文:
1. 发送端逐字符 `write((int)ch)` 只写每个字符的低 8 位,汉字"中"(0x4E2D)出去只剩 0x2D;
2. 接收端逐字节 `(char)read()` 强转拼接,不做多字节解码。

**核心思想**:正文改成**整段 UTF-8 字节**收发;原协议的 `0xFF`(消息结束符)、`0xFD`(命令标记)两个控制字节保持不变——它们不会出现在合法 UTF-8 编码中,与中文字节天然不冲突。

**四个核心点(收发各两处,客户端 + 服务器)**:

**① 客户端发送**(`ClientKernel.java` → `ClientMsgSender.run`):
```java
// 命令消息(以 0xFD 开头)与普通聊天分开组装,但正文一律"整段 UTF-8 编码":
if(msg.charAt(0) == COMMAND) {                 // 这是一条命令(如 nick/msg/file)
    byte[] b = msg.substring(1).getBytes("UTF-8"); // 去掉命令前缀,正文整体转 UTF-8 字节数组
    plain = new byte[b.length + 1];
    plain[0] = (byte)0xFD;                     // 0xFD 命令标记单独作为一个"原始字节"放在最前
    System.arraycopy(b, 0, plain, 1, b.length);// 拼成:0xFD + 正文UTF-8字节
} else {
    plain = msg.getBytes("UTF-8");             // 普通聊天:正文整体转 UTF-8 字节
}
dataOut.write(...plain...);                    // 把这一整段字节写出去
dataOut.write(MSGENDCHAR);                      // 末尾补 0xFF 作为一条消息的边界(分帧)
```
关键点:发送不再逐字符 `write((int)ch)`(那样只写低 8 位、汉字必坏),而是把正文一次性 `getBytes("UTF-8")` 成完整字节数组再写;`0xFD` 作为独立字节而非字符,不参与 UTF-8 编码。

**② 客户端接收**(`ClientKernel.java` → `ClientMsgListener.run`):
```java
// 一个字节一个字节地读进缓冲区,直到遇到 0xFF(一条消息结束)或连接断开(-1):
java.io.ByteArrayOutputStream baos = new java.io.ByteArrayOutputStream();
while((c = dataIn.read()) != MSGENDCHAR && c != -1)
    baos.write(c);                             // 先把整条消息的原始字节"攒齐",不做任何解码
if(c == -1) break;                              // 对端断开则退出(见顺带修复②)
...
String s = new String(bytes, "UTF-8");         // 整段字节"一次性"UTF-8 解码,多字节汉字才能正确还原
```
关键点:接收不再逐字节 `(char)read()` 强转拼接(那样把每个字节当一个字符,多字节汉字被拆坏),而是攒满一条消息后 `new String(bytes,"UTF-8")` 整体解码。

**③ 服务器发送**(`ConnectedClient.java`,发送线程 `run`):
```java
dataOut.write(...(toSend.getBytes("UTF-8"))); // 整条消息整体 UTF-8 编码后写出
dataOut.write(MainServer.MSGENDCHAR);          // 末尾补 0xFF 分帧
```

**④ 服务器接收**(`ConnectedClient.java`,接收线程 `run`):
```java
while((c = dataIn.read()) != 0xff && c != -1)  // 同样:攒字节直到 0xFF 或断开
    ...
// 若首字节是 0xFD 说明这是命令,跳过它再解码正文;否则整段解码:
String body = isCommand ? new String(raw, 1, raw.length - 1, "UTF-8")  // 去掉 0xFD 命令字节
                        : new String(raw, "UTF-8");                     // 普通聊天整段解码
```

**对照"改之前错在哪"**:原程序发送逐字符 `write((int)ch)` 只写低 8 位、接收逐字节 `(char)read()` 强转,多字节汉字两头都被破坏;现在两端统一为"整段字节 + 单次 UTF-8 编解码",从根上解决,且收发对称。

**随本条顺带完成的同段代码修复**(计入本条,不另计):
- 发送空消息时 `charAt(0)` 越界崩溃 → 加空串保护:`ClientKernel.sendMessage()` 开头 `if(str==null || str.length()==0) return;`;
- 客户端/服务器接收线程在对端断开(`read` 返回 -1)后无限空转死循环 → 读到 -1 干净退出:客户端 `ClientMsgListener` 里 `if(c==-1) break;`、服务器 `ConnectedClient` 里 `if(c==-1){ cc.dropClient(); return; }`。

**说明(避免与功能十三混淆)**:当前代码的发送/接收里,UTF-8 编码之后、0xFF 分帧之前还夹了一层 `CryptoUtil.wrap/unwrap`——那是**功能十三(消息加密)**后来套在外面的,不属于本条。功能一本身只是 `getBytes("UTF-8")` / `new String(...,"UTF-8")` + 0xFF 分帧;加 `-Dchat.plain=true` 启动时加密层透传,即为本条原貌。

**工程配套**:`ChatClient.java` 源文件由 GBK 转为 UTF-8,全工程统一 UTF-8 编码;编译命令为 `javac -encoding UTF-8`(否则源码里的中文字面量本身就会损坏)。

## 功能三:客户端界面显示在线用户列表 —— 对应文档要求① 【已完成 ✅】

**核心思想**:服务器在名单变化时**主动推送**完整在线名单;为让客户端能把"推送"和"聊天"分开,新增第三个控制字节 `0x01`(服务器推送标记)。客户端凭首字节即可无歧义分流,不进聊天区。

|     字节     |       方向       |             含义             |
|:-----------:|:----------------:|:----------------------------:|
| 0xFF        | 双向             | 消息结束符                   |
| 0xFD        | 客户端→服务器    | "这是命令不是聊天"           |
| 0x01(新增)  | 服务器→客户端    | "这是推送不是聊天"           |

**① 协议层 —— 新增 0x01 服务器推送标记**:推送格式 `0x01 + "USERLIST" + 空格分隔的昵称`。三点设计考虑:①`0x01` 控制字符用户打不出来,且聊天广播都带"昵称:"前缀,客户端凭首字符即可无歧义区分推送与聊天,**无法被聊天内容伪造**;②昵称由 StringTokenizer 按空白切分而来必不含空格,用空格分隔安全;③`0x01` 的 UTF-8 编码就是单字节 0x01,与 0xFF 结束符/0xFD 命令标记互不冲突。

**② 服务器发送 —— `ConnectionKeeper.broadcastUserList()`**(`ConnectionKeeper.java` → `broadcastUserList`):
```java
public synchronized void broadcastUserList() {           // synchronized:遍历连接池时防并发改动(功能二 #9)
    StringBuffer sb = new StringBuffer();
    sb.append(MainServer.PUSHMARKER).append("USERLIST");  // 拼头:0x01 + "USERLIST"
    for(...) sb.append(' ').append(client.getNick());     // 再把每个在线昵称用空格拼上
    String msg = sb.toString();
    for(...) client.sendMessage(msg);                     // 推给连接池里每一个在线客户端
}
```
名单一变就调它——连接加入、连接移除、`/nick` 改名成功、`/register` 注册成功、`/verify` 验证成功/失败、验证超次昵称被重置,共 6 类触发点(散落在 ConnectionKeeper 的加人/移除与 BroadcastCommandParser 的改名/注册/验证处)。

**③ 客户端接收 —— `handlePush("USERLIST")`**(`ChatClient.java` → `handlePush`):
```java
if(cmd.startsWith("USERLIST")) {
    userModel.clear(); online.clear();
    StringTokenizer st = new StringTokenizer(cmd.substring(8)); // 跳过 "USERLIST" 这 8 个字符

        userModel.addElement(u);  // 填进右侧"在线用户"列表(JList 的 model)
```

## 功能四:两个用户之间的私聊 —— 对应文档要求② 【已完成 ✅】

**核心思想**:私聊与群聊**走同一条消息通道**,唯一区别在服务器端的"路由"——群聊 `broadcast()` 发给连接池里所有人,私聊 `sendTo()` 只发目标一个人;发送方另收一条回显。

> 演进说明:v1.0 时私聊是"紫色 `[私聊]` 文本前缀 + 手敲 `/msg`";功能九(会话式界面)把它升级为**独立会话窗口 + 结构化推送 PM/PMSENT**——双击即开会话、在会话里输入普通文字自动路由给对方,不再手敲命令。以下为当前真实代码。

**① 界面入口 —— 双击在线用户开私聊会话**(`ChatClient.java` → userList 的 MouseAdapter):
```java
if(e.getClickCount() == 2) {                               // 双击在线用户
    Object v = userList.getSelectedValue();
    if(v.toString().equalsIgnoreCase(txtNick.getText())) return; // 不能和自己开会话
    openConv(v.toString());   // 打开/切换到与他的独立私聊会话窗格(CardLayout)
}
```
在私聊会话里输入普通文字,发送时自动转成 `msg 目标 正文` 命令(0xFD 命令标记 + 正文),与手敲 `/msg` 完全一致。

**② 服务器路由 —— `msg()`**(`BroadcastCommandParser.java` → `msg`):
```java
long id = MainServer.nextMsgId();                          // 给这条私聊分配唯一 id(供已读/撤回)
boolean found = cc.sendTo(user, "0x01 PM " + id + " " + cc.nick + " " + body); // 只投目标一人
if(found)
    cc.sendMessage("0x01 PMSENT " + id + " " + user + " " + body);  // 找到才给发送方回显
else if(ds.isRegistered(user))
    ds.addOffline(...);                                    // 对方离线但已注册→存离线消息(功能十六)
```
入口处还做了健壮性拦截:空内容回用法提示、禁止私聊自己(大小写变体也拦)、被禁言直接拦(功能十八)。

**③ 单点投递 vs 群发 —— `sendTo()`**(`ConnectionKeeper.java` → `sendTo`):
```java
public synchronized boolean sendTo(sender, user, msg) {
    for(遍历连接池)
        if(user.equalsIgnoreCase(receiver.nick)) {         // 按昵称(忽略大小写)找目标
            receiver.sendMessage(msg);                     // 只调这一个人的发送队列
            found = true; break;
        }
    if(!found) sender.sendMessage("Unable to find user " + user); // 找不到只提示,不误导
    return found;   // 返回是否找到,msg() 据此决定要不要给发送方回显
}
```
对比群聊 `broadcast()` 是对池内**每个人**调 sendMessage;私聊只对匹配昵称的**那一个**。发送方回显仅在 `found` 为真时才发,避免"发给不存在的人却显示已发出"。

**④ 客户端渲染 —— PM / PMSENT 两类推送**(`ChatClient.java` → `handlePush`):
```java
} else if(cmd.startsWith("PM ")) {          // 收到对方私聊 PM <id> <发送者> <正文>
    ensureConv(sender).putMsg("m"+id, OTHER, sender, body);  // 进"对方"的会话窗格,靠左气泡
    if(sender.equals(currentConv)) ck.sendMessage("/read " + id); // 正看着就立即回执已读
} else if(cmd.startsWith("PMSENT ")) {      // 自己发出的回显 PMSENT <id> <目标> <正文>
    ensureConv(target).putMsg("m"+id, SELF, 我, body + " · 已送达"); // 靠右气泡 + 状态
}
```
两端都按会话名(对方昵称)把消息路由进对应窗格,群聊与各路私聊互不干扰。

## 功能五:个人信息的修改、保存和查询 —— 对应文档要求④ 【已完成 ✅】

**核心思想**:填实参考程序 DataSource/FileDataSource 预留的 info 空壳(原 addInfo 假返回 true、getAllUserInfo 返回假数据)。数据是"**(用户, 字段, 值)** 三元组",`(用户,字段)` 作逻辑主键、**全服共享一份**——所以能查他人;重复设置即覆盖;每次改动全量重写落盘。

**① 命令入口 —— `setInfo()` / `getInfo()` / `delInfo()`**(`BroadcastCommandParser.java` → `setInfo`):
```java

if(field.equalsIgnoreCase(FRIEND_FIELD) || field.equalsIgnoreCase(AVATAR_FIELD)) { 拒绝; return; } // friends/avatar 是系统保留字段,屏蔽防误改
StringBuffer sb = new StringBuffer();
while(strTok.hasMoreTokens()) sb.append(strTok.nextToken() + " "); // 值取"剩余全部 token"拼接 → 允许含空格
ds.addInfo(cc.nick, field, sb.toString().trim());                 // 交数据层落盘
```
关键:值用剩余 token 全部拼接,**修复原程序只取 3 个词、含空格的值被截断**的问题;字段和值都支持中文。

**② 数据层存取 —— `addInfo()` / `getInfo()`**(`FileDataSource.java` → `addInfo`):
```java
public boolean addInfo(user, field, data) {
    removeFromList(user, field);                // 先删同 (user,field) 旧值 —— 这就是"覆盖修改"
    infoList.add(new Info(user, field, data));  // 内存 LinkedList 加一条新三元组
    rewriteInfoFile();                          // 再全量重写落盘
}
```
`getInfo(user,field)` 遍历 `infoList` 按 (user,field) 忽略大小写匹配;`getAllUserInfo(user)` 收集某人全部字段(供 `/getinfo` 展示,同样屏蔽 friends/avatar 保留字段)。

**③ 持久化 —— `rewriteInfoFile()`**(`FileDataSource.java` → `rewriteInfoFile`):
```java
OutputStreamWriter osw = new OutputStreamWriter(fo, "UTF-8"); // ★统一 UTF-8:修复原 writeBytes 低 8 位截断→中文进文件就坏(与网络中文同根源)
for(Info inf : infoList) w.write(inf.user + " " + inf.field + " " + inf.data + "\n"); // 全量重写 ~/.mihalychat/info
```
用"**直接覆盖写**"而非"临时文件+renameTo"(Windows 上 renameTo 到已存在文件会失败,导致修改/删除不生效——顺带修复的坑)。服务器启动时 `readFiles()` 逐行回填:前两个空格切出 user/field、行剩余整段作值。

**为什么能查他人**:数据按用户名索引、全服共享同一个 DataSource,`/getinfo 别人` 直接查得到(信息查询按文档④本就对所有人开放)。

## 功能六:增加好友与上线提醒 —— 对应文档要求③ 【已完成 ✅】

**核心思想**:好友名单**不新建存储**,而是复用功能五——存成一条保留字段 `(用户, "friends", "名字1 名字2 ...")`。上线提醒是"**谁把我加为好友,就通知谁**",方向与文档要求一致。

**① 加/删好友 —— `addFriend()` / `delFriend()`**(`BroadcastCommandParser.java` → `addFriend`):
```java

if(containsToken(cur, name)) { 已在列表; return; }             // 查重:containsToken 按空格切分逐个比对
ds.addInfo(cc.nick, FRIEND_FIELD, cur==null ? name : cur+" "+name); // 追加,写回同一存储(复用功能五)

```
`delFriend()` 同理:用 StringTokenizer 把好友串按空格重组、去掉目标。`isOnline()` 对照 `ConnectionKeeper.users()` 连接池快照判断在线。

**② 上线提醒 —— `notifyFriendOnline()`**(`BroadcastCommandParser.java` → `notifyFriendOnline`):
```java
private void notifyFriendOnline(ConnectedClient cc) {          // cc = 刚确立昵称的人
    for(遍历所有在线用户 u) {
        if(u == cc) continue;
        if(containsToken(ds.getInfo(u.getNick(), FRIEND_FIELD), cc.nick)) // u 的好友串里有没有 cc?

```

## 功能七:用户之间传输文件 —— 对应文档要求⑦ 【已完成 ✅】

**核心思想**:文件经 **Base64 编码后当普通消息发**,服务器按目标转发、**不另开数据端口**。Base64 是纯 ASCII,不会撞 0xFF/0xFD/0x01 任何协议字节,所以能直接复用既有的消息边界与收发线程——最小改动获得可靠传输。

> 演进说明:v1.0 功能七是"单包 Base64 `file` 命令"(代码仍在 `fileTransfer`/`receiveFile`,兼容保留);功能十二把"文件"按钮升级为**分块传输**(fstart/fchunk/fend + 进度条,解除大小限制)。以下讲功能七本身的单包核心。

**① 发送端 —— Base64 + 走消息通道**(`ChatClient.sendFile` → `ClientKernel.sendFile`):
```java
byte[] bytes = Files.readAllBytes(file);                  // 读文件字节
String b64 = Base64.getEncoder().encodeToString(bytes);   // 编成纯 ASCII 字符串
ck.sendFile(target, fname, b64);                          // ClientKernel.sendFile:
// → cms.addMessage("0xFD" + "file 目标 文件名 base64串");  // 0xFD 命令标记 + 0xFF 结尾,当普通命令消息发
```
右侧选中人=私发、未选=群发 `*`;文件名空格转下划线以适配空格分隔的命令格式。

**② 服务器转发 —— `fileTransfer()`**(`BroadcastCommandParser.java` → `fileTransfer`):
```java
String push = "0x01 FILE " + cc.nick + " " + fname + " " + b64;   // 重组为 0x01 推送
if(target.equals("*")) for(池内每人 u!=cc) u.sendMessage(push);   // 群发:除发送者外逐个投递
else cc.sendTo(target, push);                                     // 私发:sendTo 单点投递
```
服务器**不解码文件内容,纯转发**;被禁言拦截(功能十八)、不能发给自己、找不到目标提示。

**③ 接收端 —— `receiveFile()`**(`ChatClient.java` → `receiveFile`):
```java
String fname = ...substring(...).replace('\\','_').replace('/','_'); // ★文件名来自网络,剥掉路径分隔符防目录穿越
byte[] data = Base64.getDecoder().decode(...);              // 解码回字节
Files.write(downloadTarget(fname).toPath(), data);          // 存到 ~/ChatDownloads(重名加时间戳前缀)
ensureConv(pane).putMsg(..., OTHER, sender, buildFileHtml(out, fname)); // 会话里展示
```
分流入口是 `addMsg()` 的 0x01 首字节 → `handlePush("FILE")`。安全点:`..\evil.txt` 被改写成 `.._evil.txt`,写不到上级目录。文件不占独立端口、不开新线程,靠的是消息协议本身对"任意长度、ASCII 安全内容"的承载能力。

## 功能十:局域网多机部署与服务器自动发现 【阶段一 · 已完成 ✅】

让程序真正跑在同一局域网的多台电脑上,且"零配置"连接:

- **部署基础**(单独不足一条,作为前置并入):服务器启动时自动检测并打印本机局域网 IP;打包可执行 JAR + 启动脚本,目标机只需 JRE 双击即跑;整理防火墙放行操作说明;
- **自动发现**(本条核心):客户端加"扫描局域网"按钮——发 UDP 广播探测包,服务器监听广播端口并应答自己的 IP/端口,客户端自动填入 Host 框,连 IP 都不用查。

核心技术:**UDP DatagramSocket 广播**(为项目引入 TCP 之外的第二种 socket)。工作量:中。
答辩价值:★最高——TCP/UDP 对比、广播与单播原理,纯正计网内容;"为什么发现用 UDP、聊天用 TCP"是送分问答。

**实现简述**:服务器 DiscoveryResponder 线程在 UDP 3600 端口 `DatagramSocket.receive()` 阻塞等广播,收到 `CHAT_DISCOVER` 即向来包的源地址回 `CHAT_SERVER <TCP端口>`;客户端"扫描局域网"按钮**枚举本机每块网卡的子网广播地址逐个发探测包**(再补一份 255.255.255.255 兜底,`setBroadcast(true)`),从**应答包的源 IP** 得知服务器地址,自动填入 Host/Port。服务器启动时枚举 NetworkInterface 打印全部本机 IPv4;`build-jar.bat` 产出 chat-server.jar / chat-client.jar 双击即跑。
**多网卡修复(实际使用中暴露)**:早期只往 255.255.255.255 发单个广播,在装有 VMware/虚拟机(多出 192.168.x.1 等虚拟网卡)的机器上,广播只会从其中一块网卡发出、往往不是连着热点的那块,导致"手填 IP 能连、扫描却发现不了服务器"。改为遍历所有网卡的子网定向广播后解决。另需在服务器机放行防火墙 UDP 3600 入站(演示部署说明已列)。
**验证**:集成测试发 UDP 探测收到 `CHAT_SERVER 3500` 应答;JAR 冒烟启动服务器+双客户端全部正常。

## 功能十一:心跳保活与断线自动重连 【阶段一 · 已完成 ✅】

- **心跳保活**:客户端定时发心跳,服务器超时未收到即判死连接并清理(拔网线、笔记本睡眠等"半死连接"只有多机环境才暴露);
- **自动重连**:客户端察觉断线后按原昵称自动重连并恢复会话(注册用户自动重新验证),不再需要手动 Connect;重连期间的消息进入待发队列,恢复后补发。

核心技术:应用层 keepalive 协议 + 重连状态机。工作量:中。
答辩价值:★高——"TCP 自带 keepalive(默认 2 小时)为何还要应用层心跳(秒级感知)"。

**实现简述**:客户端 Heartbeat 线程每 5 秒发 `/hb <序号> <时间戳>`,服务器原样推回(`0x01 HB ...`),一包两用——客户端 15 秒收不到应答判定服务器死亡并触发 connectionLost;应答往返差即 RTT(存于 lastRtt,阶段二面板的数据源)。服务器侧 ConnectedClient.lastHeard 在收到任何消息时刷新,HeartbeatSweeper 线程每 10 秒清扫超 30 秒无消息的"半死连接"。断线后客户端进入重连状态机:每 3 秒重试(手动 Connect 即取消),成功后自动取回昵称——若昵称已注册,凭发送 /register、/verify 时缓存的密码**自动完成身份验证**;断线期间输入的消息进待发队列,恢复后按序补发。
**验证**:集成测试实测 RTT≈32ms;/exit 模拟断线后自动重连成功、昵称取回、在线名单恢复。

## 功能十二:文件分块传输与进度显示 【阶段一 · 已完成 ✅】

大文件切块编号传输、接收端重组校验,双方显示进度条;传输不再阻塞聊天消息,解除 20MB 上限。随本条的协议升级一并引入**消息唯一 ID**(功能十九/二十的前置)。

核心技术:**自定义应用层协议**(分块/序号/重组/完整性)。工作量:中偏大。
答辩价值:★次高——"自己设计应用层协议"是课设的灵魂加分项,可讲分块大小取舍与交织发送。

**实现简述**:发送端把文件切成 64KB 块,协议三连——`/fstart 目标 tid 文件名 大小 块数` → 逐块 `/fchunk tid 序号 Base64` → `/fend tid`;服务器不落地文件,按 tid 查路由表纯转发(`0x01 FSTART/FCHUNK/FEND` 推送);接收端 FSTART 建档开流、FCHUNK 顺序落盘(TCP+FIFO 队列保证有序)、FEND 校验大小定稿展示。**交织发送**:发送线程在发送队列有积压时等待,期间输入的聊天消息插进块间隙,传输不再阻塞聊天;双侧进度按百分比原位刷新(聊天窗格的带键行更新)。上限从 20MB 放宽至 200MB。随本条协议升级引入**消息唯一 ID**:群聊 `0x01 MSG <id> <昵称> <正文>`、私聊 PM/PMSENT 均带 id,消息首次可寻址(阶段二回执/撤回的前置)。
**验证**:300KB 随机内容文件走完整加密链路分块传输,接收方落盘字节与原文件完全一致,双侧窗格进度与定稿消息正确。

## 功能二十一:本地聊天记录 【阶段三 · 已完成 ✅】

客户端把各会话历史按会话分文件落盘,重开程序自动回填各窗格;支持右键清空单个会话记录;记录带时间戳。

核心技术:客户端文件存储 + 启动回填(与会话式界面联动)。工作量:中。与功能十六(离线消息)互补:一个在服务器端管"没收到的",一个在客户端管"收到过的"。

**实现简述**:**按登录身份隔离**——历史存于 `~/.chattool-history/<昵称Base64>/<会话名Base64>.txt`(URL-safe Base64 安全承载中文/#等)。登录并确立昵称后(connect / syncNick 确认)才 loadOwnerHistory() 载入"我这个身份"的记录;身份切换时先清空面板再载入,同身份自动跳过。ClientHistory 每次消息增改即把该会话全部非系统消息序列化重写(字段用 0x01 分隔:类型 发送者 时间 rid 正文),整体重写总能反映撤回/已读后的最新态;系统提示不入历史。会话列表右键"清空聊天记录"清内存+删文件。
**为何按身份隔离**(修复一个真实 bug):早期按机器全局存,同机开两个客户端(如 Brendan 与 user1)会写同一批文件、按各自视角标 SELF/OTHER 互相覆盖,导致重开后"别人的消息显示成自己发的"。按昵称分目录后各存各的,互不污染。
**GUI 实测暴露的第二个 bug —— 历史消息被新消息覆盖(已修复)**:消息 ID 由服务器分配,而服务器重启后 ID 计数从头重排。回填的历史消息带的是上一会话的旧 ID,新会话发的消息又拿到相同 ID,气泡按 ID 索引更新时,新消息就"命中"并覆盖掉同 ID 的那条历史(现象:重连后发一条消息,把历史里 alice 的第一条顶掉了)。根因是"消息 ID 可跨会话复用"与"气泡按 ID 原位更新"两个机制的冲突。修复:回填历史时把每条的 rid 置 -1、且不登记进 ID→气泡 的索引表,历史消息从此只读、不可被后续同 ID 消息定位覆盖;实时消息的原位更新(文件进度/撤回/已读)不受影响。
**验证**:同机两身份历史隔离测试——重开 Brendan 恰好一条 SELF(自己)一条 OTHER(user1),不串;群聊/私聊落盘、清空删文件;历史覆盖专项测试——回填一条 alice 历史后,重启服务器发同 ID 新消息,历史条目仍在、新消息正常追加。无界面测试通过。

## 功能二十二:用户头像 【阶段三 · 已完成 ✅】

完整的头像体系:客户端选图上传(自动压缩到头像尺寸)→ 服务器按用户存储并向在线用户分发 → 列表渲染,他人上线时自动同步,客户端本地缓存。

核心技术:文件传输+个人信息存储的组合应用 + Swing 自定义渲染器。工作量:中。

**实现简述**:"设置头像"按钮选图 → 客户端缩放到 **128×128 PNG**(先铺白底再绘制,透明 PNG 才不会渲染成黑块)→ Base64(数 KB)以 /avatar 命令上传;服务器存入保留字段 avatar(与 friends 同机制,对 getinfo/setinfo 屏蔽)并广播 `0x01 AVATAR <昵称> <b64>`;三个昵称确立点调用 pushAvatars——给新上线者发所有在线者的头像、把新人的头像发给所有人(双向同步)。客户端解码后缓存到 ~/.chattool-avatars/(重启可用),在线用户列表与会话列表通过自定义 DefaultListCellRenderer 在名字前绘制 20px 头像,聊天气泡旁(仿微信)绘制 36px 头像。**未设置头像者显示默认头像**(代码用 Graphics2D 绘制的灰底白色人形 `_default.png`,首次用到时生成),不再是空白或黑图。
**验证**:A 上传后 B 实时收到;后上线的 C 自动同步到 A 的头像;未设头像时列表与气泡均显示默认人形。集成测试 2/2。

## 功能二十五:快捷输入增强 【阶段三 · 已完成 ✅】

把"发出一条内容"的操作成本压到最低:

核心技术:剪贴板 API(Transferable/DataFlavor)+ Swing DropTarget 拖放。工作量:中。

**【已完成 ✅】实现简述**:截图粘贴——输入框 keyPressed 拦 Ctrl+V,系统剪贴板 imageFlavor 有图即 consume 默认粘贴,图片转 PNG 临时文件走 sendFileChunked 发到当前会话(QQ/微信截图后一键贴发);拖拽发送——每个聊天面板(ClientHistory)各挂一个 DropTarget,javaFileListFlavor 的文件逐个发送(200MB 限);快捷键——↑ 键重填上一条(原有)、Esc 最小化隐入托盘。
**拖拽失效修复(实际使用中暴露)**:早期只把 DropTarget 挂在外层容器(centerCards)上,而真正显示聊天的 JEditorPane 有自己的拖放处理会**吞掉 drop 事件**,外层永远收不到,拖文件进去毫无反应。改为把拖放处理抽成 attachDrop(),在**每个聊天面板创建时逐一挂载**(外层容器也挂一个覆盖滚动条等空白区),拖拽才真正生效。
**验证**:GUI 实测——从资源管理器拖文件到聊天区即发送到当前会话;剪贴板/拖放为桌面交互,无界面环境不可自动断言,编译与事件链路通过。

## 功能二十六:表情系统 【阶段三 · 已完成 ✅】

不止一个选择面板,而是完整的表情机制:

- **图片表情**:内置一套经典表情,定义短码协议(如 `/微笑`),发送端转短码、接收端还原为 `<img>` 内嵌渲染;
- **自定义表情**:任意图片可收藏为自定义表情,底层复用文件传输,双端缓存。

核心技术:表情短码协议 + 资源管理与渲染。工作量:中。

**【已完成 ✅】实现简述**:内置 8 个表情(微笑/大笑/哭/生气/酷/惊讶/爱心/赞)由 **Graphics2D 代码绘制**,首次运行生成到 ~/.chattool-emotes/——每个客户端各自生成同一套,聊天只传短码文本(如 /微笑),接收端 Emotes.apply() 在气泡渲染时把短码替换为本地 `<img>`,**零流量图片表情**(这是短码协议的核心价值)。"表情"按钮弹网格面板:点内置表情插短码入输入框;"＋"收藏任意图片为自定义表情,右键面板中的自定义表情可**删除**。
**自定义表情优化**:收藏时把图片**缩放到 ≤128px 的小 PNG**、文件名加 `emote_` 前缀存入 custom/ 目录;发送/接收端据前缀把它渲染成**内嵌小表情(≤100px)**而非"[图片]（查看原图）"大图,观感与内置表情一致,且不显示传输进度。缩小的意义:①面板缩略图带缓存、打开秒开(此前存原图,可能几 MB,每次开面板都重读大图导致明显卡顿);②发送体积只有几 KB,近乎瞬发。诚实说明:自定义表情对端机器上没有,**必须把小图传过去一次**(几 KB),做不到内置表情那样的短码零流量,这是二者的本质区别。
**验证**:内置表情文件生成、短码经真实链路在对端渲染为 img 标签;自定义表情缩放为小图、以内嵌小表情显示、可右键删除。集成测试 2/2 + GUI 实测。
