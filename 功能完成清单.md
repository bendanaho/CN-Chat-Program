## 功能一:聊天支持中文(重写收发编码协议) —— 对应文档要求⑤ 【已完成 ✅】

**问题根源**:参考程序的通信协议是面向单字节 ASCII 设计的,从两个层面破坏中文:
1. 发送端逐字符 `write((int)ch)` 只写每个字符的低 8 位,汉字"中"(0x4E2D)出去只剩 0x2D;
2. 接收端逐字节 `(char)read()` 强转拼接,不做多字节解码。

**实现方案**:保留原协议的 0xFF 消息结束符、0xFD 命令标记不变(这两个控制字节不会出现在合法 UTF-8 编码中,与中文字节天然不冲突),把消息正文改为整体 UTF-8 字节收发:
- 发送:`msg.getBytes("UTF-8")` 整体写出,0xFD 命令标记单独作为原始字节写出;
- 接收:按字节攒入缓冲区直到 0xFF,再 `new String(bytes, "UTF-8")` 整体解码。

**随本条顺带完成的同段代码修复**(计入本条,不另计):
- 发送空消息时 `charAt(0)` 越界崩溃 → 加空串保护(ClientKernel.sendMessage);
- 客户端/服务器接收线程在对端断开(read 返回 -1)后无限空转死循环 → 读到 -1 干净退出。

**工程配套**:ChatClient.java 源文件由 GBK 转为 UTF-8,全工程统一 UTF-8 编码;编译命令为 `javac -encoding UTF-8`。

## 功能三:客户端界面显示在线用户列表 —— 对应文档要求① 【已完成 ✅】

**实现方案**:服务器推送 + 客户端实时刷新。

|     字节     |       方向       |             含义             |
|:-----------:|:----------------:|:----------------------------:|
| 0xFF        | 双向             | 消息结束符                   |
| 0xFD        | 客户端→服务器    | "这是命令不是聊天"           |
| 0x01(新增)  | 服务器→客户端    | "这是推送不是聊天"           |

- **协议层**:新增服务器主动推送消息,以控制字符 `0x01`(PUSHMARKER)开头,格式 `0x01 + "USERLIST" + 空格分隔的昵称`。设计考虑:①控制字符用户打不出来,且聊天广播都带"昵称:"前缀,客户端凭首字符即可无歧义区分推送与聊天,无法被聊天内容伪造;②昵称由 StringTokenizer 按空白切分而来必不含空格,空格分隔安全;③0x01 的 UTF-8 编码是单字节 0x01,与 0xFF 结束符/0xFD 命令标记无冲突。
- **服务器端**(ConnectionKeeper 新增 broadcastUserList()):名单变化的全部 6 个触发点都会向所有在线客户端推送完整名单——连接加入、连接移除、/nick 改名成功、/register 注册成功、/verify 验证成功/失败(昵称都会变)、验证超次昵称被重置。
- **客户端**(ChatClient):界面右侧(BorderLayout.EAST)新增"在线用户"侧栏(JList + JScrollPane);addMsg() 拦截 0x01 开头的推送更新列表,不进聊天区;断线时(connectionLost)自动清空列表。

## 功能四:两个用户之间的私聊 —— 对应文档要求② 【已完成 ✅】

在参考程序已有的 /msg 服务器命令基础上补齐界面入口、消息区分和回显,使私聊成为完整可用的功能:

- **界面入口**:双击右侧在线用户列表中的名字,输入框自动填好 `/msg 名字 `,光标就位直接输入内容回车即发;
- **消息区分**:私聊消息以紫色 `[私聊]` 标记显示,与群聊一眼区分(接收方看到 `[私聊] 发送者: 内容`);
- **发送方回显**:发出的私聊在自己聊天区显示 `[私聊→对方] 内容`(原程序发私聊后自己界面毫无反应);仅在对方存在时回显——sendTo() 改为返回是否找到目标,找不到时只提示 "Unable to find user",不产生误导性回显;
- **健壮性**:空内容私聊返回用法提示(原程序会发出一条空消息);禁止私聊自己(服务器端拦截并提示,大小写变体也拦;客户端双击自己名字不预填命令,双重防护)。

**实现简述**:私聊与群聊走同一条消息通道,区别只在服务器端的"路由"。客户端双击事件(ChatClient 中 userList 的 MouseAdapter)只是把 `/msg 名字 ` 预填进输入框,发送时与手敲命令完全一致:0xFD 命令标记 + "msg 目标 内容"。服务器 BroadcastCommandParser.msg() 解析出目标和正文,拼上紫色 [私聊] 标记后调 ConnectionKeeper.sendTo()——它遍历连接池按昵称(忽略大小写)找到目标 ConnectedClient,只调**这一个人**的 sendMessage() 入其发送队列(群聊的 broadcast() 则是对池内每个人调用);sendTo 返回是否找到,msg() 据此决定是否给发送者再发一条 [私聊→目标] 回执。

## 功能五:个人信息的修改、保存和查询 —— 对应文档要求④ 【已完成 ✅】

填实参考程序中 DataSource/FileDataSource 预留的 info 空壳(原 addInfo 假返回 true、getAllUserInfo 返回假数据),实现完整的个人信息存取:

- **命令**:`/setinfo <字段> <内容>` 设置(重复设置同字段即覆盖修改)、`/getinfo [用户]` 查询自己或他人、`/delinfo <字段>` 删除;字段和内容均支持中文,内容允许含空格;
- **存储**:服务器端落盘 `~/.mihalychat/info`,格式 `用户 字段 值`,内存 LinkedList + 全量重写持久化;修复原解析只取 3 个词导致值含空格被截断的问题;文件读写统一 UTF-8(原 writeBytes 低 8 位截断,中文进文件就坏——与网络中文问题同根源);
- **顺带修复**:removeUserFromFile 用"临时文件+renameTo"落盘,Windows 上 renameTo 到已存在文件会失败导致删除不生效,改为直接覆盖写。

**实现简述**:数据结构是 FileDataSource 内存中的 `LinkedList infoList`,元素为 Info{user, field, data} 三元组,(user, field) 作逻辑主键。写路径:BroadcastCommandParser.setInfo() 从命令中切出字段和值(值取剩余全部 token 拼接,故可含空格)→ ds.addInfo() 先删同键旧值再插新值 → rewriteInfoFile() 把整个 infoList 全量重写进 info 文件(UTF-8 Writer)。读路径:getInfo()/getAllUserInfo() 遍历 infoList 匹配;服务器启动时 readFiles() 逐行解析 info 文件回填内存(前两个空格切出 user/field,行剩余整段作值)。查询他人之所以能工作,是因为数据按用户名索引、全服共享一份 DataSource。

## 功能六:增加好友与上线提醒 —— 对应文档要求③ 【已完成 ✅】

- **命令**:`/addfriend <昵称>`(添加时即时反馈对方在线状态)、`/delfriend <昵称>`、`/friends`(列表并标注每人在线/离线);不能添加自己;重复添加有提示;
- **存储**:好友名单复用功能五的存储设施,存为保留字段 `friends`(空格分隔),随服务器持久化,重启不丢;`/setinfo`、`/getinfo` 对该保留字段屏蔽,防止误改和泄露;
- **上线提醒**:昵称确立的三个时机(/nick 成功、/register 成功、/verify 成功)服务器遍历在线用户,凡把此人加为好友者收到橙色提醒"你的好友 X 上线了";同名重设不重复提醒。

**实现简述**:好友名单不新建存储,而是作为功能五的一条特殊记录——用户 X 的好友存为 (X, "friends", "名字1 名字2 ...")。addFriend() 用 ds.getInfo() 取出旧串、查重(containsToken 按空格切分逐个比对)后追加、ds.addInfo() 写回;delFriend() 同理重组字符串。在线判断 isOnline() 对照 ConnectionKeeper.users() 返回的连接池快照。上线提醒的核心是 notifyFriendOnline(cc):挂在 setNick/registerNick/verifyNick 三个昵称确立点上,反向遍历所有在线用户 u,若 u 的 friends 串包含 cc 的新昵称,就单发一条橙色提醒给 u——即"谁关注了我,通知谁",而不是"我通知我的好友",方向与文档要求一致。

## 功能七:用户之间传输文件 —— 对应文档要求⑦ 【已完成 ✅】

**方案**:文件经 Base64 编码后作为 `file` 命令走既有消息通道,服务器按目标转发,不另开数据端口。设计考虑:Base64 为纯 ASCII,不会与 0xFF 结束符/0xFD 命令/0x01 推送任何协议字节冲突,复用全部既有的消息边界与收发线程设施,以最小改动获得可靠传输。

- **发送**:界面"文件"按钮 + JFileChooser;右侧列表选中人=私发,未选=群发所有人(点列表空白处取消选中);限 5MB;文件名中空格转下划线以适配空格分隔的命令格式;
- **转发**(服务器):`/file <目标|*> <文件名> <Base64>` → 组装 `0x01 FILE 发送者 文件名 Base64` 推送给目标(或除发送者外的所有人),发送者收到紫色回执;不能发给自己;找不到目标提示;
- **接收**(客户端):解码后保存到 `用户主目录\ChatDownloads\`,重名自动加时间戳前缀;**文件名来自网络,剥掉 / 和 \ 防目录穿越攻击**;聊天区显示来源与保存路径;
- **配套**:命令解析出错时的回显截断至 100 字符(否则 file 命令出错会把整个 Base64 弹回聊天区)。

**实现简述**:全链路四步。①发送端 ChatClient.sendFile():JFileChooser 选文件 → Files.readAllBytes 读入字节 → Base64 编码成 ASCII 字符串 → ClientKernel.sendFile() 拼成 `0xFD + "file 目标 文件名 base64串"` 入发送队列,按普通命令消息发出(0xFF 结尾);②服务器 BroadcastCommandParser.fileTransfer():切出三个 token,重组为推送格式 `0x01 + "FILE 发送者 文件名 base64串"`,目标为 `*` 时遍历连接池快照逐个投递(跳过发送者),否则走 sendTo() 单点投递——服务器**不解码文件内容**,纯转发;③接收端 ChatClient.handlePush() 由 0x01 首字符识别推送、"FILE" 前缀分流到 receiveFile():两次 indexOf 切出发送者/文件名/Base64,解码为字节写入 ChatDownloads;④文件名先剥掉 / 和 \(防目录穿越),重名加时间戳。文件不占独立端口、不需要新线程,靠的是消息协议本身对"任意长度、任意内容(ASCII 安全)"的承载能力。私发文件接收方 Base64 解码与原文一致(含中文内容)、旁观者收到 0 条;群发 2 人各收到 1 份、发送者不回环;无界面测试验证客户端落盘内容无损、`..\evil.txt` 穿越文件名被改写为 `.._evil.txt` 且上级目录未被写入。

## 功能八:聊天内容支持图片等多媒体 —— 对应文档要求⑥ 【已完成 ✅】

**方案**:骑在功能七的文件传输上——聊天区本就是 JEditorPane 的 HTML 渲染(参考程序既有设计),收到的文件若为图片(png/jpg/jpeg/gif),保存后直接以 `<img src="file:...">` **内嵌显示在聊天区**,接收方无需任何操作即可看图;非图片文件则显示保存路径。

**实现简述**:传输部分与功能七完全共用(图片就是文件),差异只在接收端最后一步:receiveFile() 保存完文件后检查扩展名,是 png/jpg/jpeg/gif 之一就不显示"已保存到..."的文本,而是拼一条含 `<img src="本地文件URI">` 的消息交给 addMsg()。聊天区 ClientHistory 继承自 JEditorPane("text/html"),其 HTML 渲染引擎看到 img 标签会自动加载本地图片文件显示——这正是参考程序把聊天区设计成 HTML 的能力兑现,一行标签即完成"多媒体支持"。

**验证**:无界面测试生成真实 1×1 PNG 走完整接收链路,聊天区 HTML 出现 `<img>` 标签且图片落盘成功;GUI 下两客户端互发图片肉眼可见。
